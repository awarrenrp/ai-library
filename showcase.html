Rippling AI × Slack rendering spec

How the Slack agent renders every type of response — answers, typed objects, tables, confirmations, async status, handoffs, and errors. Each pattern is paired with the Block Kit it produces and a faithful Slack-side render of what the user sees.

Scopeconvenience case SurfaceSlack web + desktop + mobile APIBlock Kit (Apr 2026)
§01

Block Kit reference

Slack's Block Kit is the only rendering system available — no custom CSS, no React, no canvas. These are the blocks the spec uses.

P01

Plain Q&A answer

~80% of traffic

The user asks a factual question the AI can answer from Rippling context with no action needed.

When
Read-only question with a single direct answer. No state change. No clarification needed.
Design intent
One direct answer. Conversation-length, not document-length. ≤150 words. Short citations (policy name, updated date) stay inline in the same paragraph as the answer (same body style as the rest of the sentence; policy as a normal link), not a separate grey strip. Feedback buttons always present.

P02

Typed object reference

card block

The response refers to a specific Rippling entity — employee, device, policy, request. Formatted text (Slack rich_text-style layout below) is the default reference surface in this spec: the same facts as the Block Kit card, tuned for scanability in long threads. Rendered is the optional graphical card presentation.

When
A named entity is the answer. The user needs to verify they're looking at the right thing and have one path to see more.
Design intent
Name + 3 key attributes + one link. Not the full RQL card. Enough to verify, not to read. Lead with formatted text in the spec and in product copy; card blocks where Block Kit and client support are strongest.

P03

Table / list result

table block · 1 per msg

A list of records — search results, open tickets, PTO requests, devices.

When
2+ records of the same type. The user wants to scan and pick or just see scope.
Design intent
State total_count in the opening rich_text only. Render up to 5 rows in the table. When there are more rows, rely on the View all / Open in Rippling actions for the full list — do not add a grey context line duplicating the count or an “Open all in Rippling” sentence above the primary CTAs, and do not add a Show more link under the table.

P04

Verify-and-confirm

core action pattern

The AI is about to take an action on the user's behalf. It pre-fills everything it knows from Rippling; the user confirms or corrects. Use this pattern only when the action is on the stay-in-Slack list.

When
The AI has enough Rippling context to assemble the action with no more than 2 clarifying questions.
Design intent
Three components: (1) what the AI understood, (2) a card showing pre-filled values, (3) Confirm, Open in Rippling (preview), and Cancel in the prototype. The user’s job is to confirm or correct.

P05

Action execution — working → completion

in-place + CTAs

When the user confirms an action that takes more than a beat (creating a ticket, submitting a request), the agent appends a working message (bold gerund + muted line + Open in Rippling button). When the job finishes, that same message is updated in place to an async_status-style completion: plain text with ✅, green Open in Rippling, and outline View … — no green success panel around the copy.

When
The user clicked Confirm on a Pattern 4 verify_confirm. The agent commits to a side-effecting action that takes a few seconds.
Design intent
Don’t make the user wonder if the action took. Acknowledge the commit, leave a paper trail, and ping back with the result and a link the moment it’s ready.
Prior-turn CTAs
Buttons on the verify-and-confirm card (Confirm, Open in Rippling, Cancel) disappear once the agent’s next message is posted. Slack does not keep stale actions live on older messages; the thread advances and only the latest turn carries interactive elements.
SLACK · #ask-rippling · success path + failure example
P06

Handoff to Rippling

use sparingly

The deliberate routing decision when Slack is not the right surface for the next step. See Stay in Slack vs Open in Rippling for the full decision rules and capability matrix.

When (only these three)
(a) content too rich for Slack — chart, full doc, >5-row table; (b) action requires full context to be seen before confirming; (c) data collection needs >~3 fields the AI cannot pre-fill.
Design intent
Frame as routing, not failure. "Better in Rippling" — never "I can't do this." Always carry pre-filled context through the deep link.
SLACK · DM with Rippling AI
P07

Error / system state

alert block

Something went wrong or is blocked — identity mapping failed, permission denied, action gated by policy.

When
Identity not found, permission denied, action policy-blocked, step-up required, service error.
Design intent
State what happened. State what to do next. Never a dead end.

Severity

SeverityUse
warningSoft block, user has a path forward (link account, verify identity).
errorHard block, requires admin action or escalation.
infoInformational, not a failure (feature not in Slack — use Rippling).

P08

Thinking, latency & acknowledgement

shimmer + tiers + answer

Older specs assumed a 1–3s model turn; Rippling-scale agent work often sits around ~66s p50 in practice. The UI uses two latency tiers: a short thinking state, then an explicit “I’m on it” line in the timeline until the real reply lands.

Tier 1 — thinking
Purple-gradient agent name plus rotating status line (Organizing…). Implemented once as ThinkingShimmerRow and reused for every in-flight LLM path (channel, DM, app tab, thread composer, typing strip). Runs only for the first 5–10s (prototype: 8s) — not for the full backend wait.
Tier 2 — still working
If the response is not ready when tier 1 ends, post a single muted ack-style line in the message list (roman body, not italic): I’m on it — still working on this. I’ll post the full reply here when it’s ready. The agent adds a ⏳ reaction to that turn’s user prompt so the thread signals work in flight. Remove the reaction when the final answer (or error) arrives.
Then
Structured Block Kit answer, verify-and-confirm, or error — same as other patterns. Domain-specific instant acks (e.g. “Looking that up…”) can still precede tier 1 when product requires it.

Three-state flow

Each stage is a distinct Slack message row. Tier-1 disappears the moment tier-2 starts; tier-2 disappears the moment the real answer posts. The channel is never left empty.

P09

Channel context — public vs DM

privacy gate

Same question, two surfaces. Public channels never reveal sensitive info; the agent acknowledges the topic and routes the user to a DM where the full answer is safe to share.

When
User asks for compensation, payroll, performance, leave reason, address, or any individual’s personal record — in any channel that isn’t a DM with the agent.
Design intent
Don’t leak. Don’t scold. Acknowledge the ask, explain the surface, and offer a one-click path to continue safely.

Task completion signals

prototype chrome

During tier-2 long waits, the prototype adds a ⏳ reaction on the user’s prompt (see Pattern 8). After a slack_complete action finishes, the in-thread completion message is Pattern 5. The React prototype also adds a ✅ reaction on the user’s prompt and unread-style badges — documented here as app chrome, not Block Kit.

Reaction
The agent adds a ✅ emoji reaction to the user message that immediately preceded the verify card (the prompt that started the action).
Unread-style badges
Activity in the left icon rail shows a red count. The relevant sidebar row (#channel, DM, or app) shows the same count until the user opens that destination.

P12

Long-form answers & per-section collapse

show more / show less

Some agent replies read like mini-reports: multiple headings, tables, export actions, and long numbered insights. Without structure, the message dominates the channel and buries everything below it. Treat each logical section (or very long paragraph) as its own collapsible unit with Show more / Show less — not a single toggle for the entire message.

Interactive demo

Sections that exceed the threshold start collapsed. Each “Show more” toggle is independent — expanding one does not open the others.

SLACK · long-form report · per-section collapse
Problem
One monolithic rich_text (or stacked blocks) that exceeds ~1–2 viewport heights pushes the thread’s context away, feels overwhelming on mobile, and hides the fact that some sections are optional detail.
Intent
Keep the title + first paragraph + first data block visible by default. Defer dense subsections behind Show more so scanning stays fast; power readers expand only what they need.

Granularity — what gets its own toggle

UnitCollapse whenNotes
Major sectionSection body (after heading) exceeds threshold, or contains a large table.Prefer one toggle per H2-equivalent section (e.g. “Equity alignment by performance”), not one for the whole report.
Long paragraph / listA single subsection (e.g. numbered insight #3) exceeds threshold while siblings stay short.Nested Show more under the same message is OK; avoid more than ~3 open expansions by default.
Table + takeawayTable has many rows or wide columns; keep header row + first 2–3 rows visible, remainder behind toggle.Do not collapse the only table in a “here are your numbers” answer without showing at least one data row.
Footer blocksReport pill, sources, feedback stay always visible.Same as other patterns: citations and feedback are not behind collapse.

Interaction & copy

RuleSpec
Collapsed previewEnd the visible slice with a natural break (sentence or list item), then ellipsis if mid-thought is unavoidable (see interactive demo below). Inline code may truncate with ellipsis inside the chip.
ControlPrimary-style text link (Show more / Show less), left-aligned directly under the block it controls — not a full-width secondary button unless product explicitly requires it.
StateIndependent per section: expanding one block does not auto-expand others. Remember state for the user’s session in the client; new messages start collapsed per spec defaults.
StreamingWhile tokens stream in, do not flip collapsed/expanded repeatedly; apply collapse rules once the section is finalized (or on first stable boundary).

Block Kit mapping

Slack has no native “accordion” block. Encode each collapsible slice as its own rich_text (or table + optional rich_text summary) plus an actions row with a single button whose label alternates Show more / Show less. The React prototype uses local state until Block Kit actions are wired.

P13

Thread-scoped conversation

prototype · channel

In public channels the prototype treats the agent as thread-first: the user’s @mention on the main timeline becomes the parent of a dedicated thread, and every model reply stays under that parent. The agent does not see other channels, the main timeline above the parent, or DMs unless the user brings that text into the thread.

Auto-thread on @mention
When someone tags the agent from the channel timeline (with a Rippling @mention), the first bot activity is stored as replies on that parent message and the UI opens the thread panel (queueMicrotask(() => onOpenThread(parent)) in src/components/ChannelView.tsx). Same pattern for errors and async completion updates on that turn.
Outside-thread asks
If a follow-up in the thread reads like “use what was said in #another-channel”, “the message above in the channel”, “outside the thread”, etc., the prototype prepends a short scope notice ahead of the LLM reply: paste or describe what you mean — implemented as ThreadScopeNotice + buildThreadScopeMessage in src/components/ThreadScopeNotice.tsx, gated by userRequestsOutsideThreadContext in src/utils/outsideThreadContext.ts from src/components/ThreadPanel.tsx.
Design intent
Make the boundary explicit without sounding broken: the user should understand that Slack/Rippling only passed the thread transcript into the model, not guess why the answer is shallow.
Code map
Auto-open thread: ChannelView.tsx (append pending, bot reply, error paths). Scope gate + notice: ThreadPanel.tsx (shouldTriggerBot path) · outsideThreadContext.ts · ThreadScopeNotice.tsx.

§04

Open · data collection > 3 fields

unresolved

The AI tries to pre-fill a verify-and-confirm card but cannot infer enough, and the action genuinely needs >3 fields. Modals are off the table — what happens?

OptDescriptionTrade-off
ASequential questions inline, build the card progressively, present confirm when complete.Natural for ≤3 questions. Tedious past that.
BRoute to Rippling via Pattern 6 immediately when the AI can't pre-fill all but ≤3 fields.Clean, honest. May feel like a frequent bailout if pre-fill is shallow at launch.
CProgressive pre-fill — show card with what's known, mark unknown fields as "needs your input", ask one at a time.Most transparent. Complex to implement. Long-term ideal.
Proposed direction · not confirmed
Option B as launch default, target <15% handoff rate for common actions as an AI quality metric. Improve context pre-fill over time. Option C is the long-term ideal. Decision needed: Product + Eng + Design after launch data.
§06

Open · multi-agent routing & orchestration

unresolved

The prototype wires one default agent per channel and polite out-of-domain replies (see Pattern 10 · #scheduling mock). It does not define how agents cooperate across channels or how multi-intent asks should fan out.

1. Cross-agent routing
Should Rippling AI be able to delegate a thread or payload to @IT Helpdesk, @Scheduling, or another specialist — automatically, with a deep link, or only by telling the user where to go?
2. Single orchestrator
When one user message mixes domains (e.g. rooms + hardware), should a single orchestrator split work across specialists, or should the product force separate @mentions / threads?
Open question · not in prototype
The Slack mocks below are exploratory scenarios only — they are not implemented behavior. Product, Eng, and Design need to pick defaults for routing, permissions, and threading before we treat any of this as spec.
